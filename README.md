# Бот аренды и продажи Steam-аккаунтов (Playerok)

Принимает оплаченные заказы, занимает аккаунт со склада и выдаёт прямо в чат
сделки логин, пароль и код Steam Guard из maFile. Аккаунт можно **сдавать в
аренду сколько угодно раз**: по окончании срока он сам возвращается на склад.

## Как работает аренда

Ключевой момент: **maFile остаётся у вас**. Арендатор получает не файл, а
одноразовые коды по запросу. Без кода в Steam не войти — поэтому когда срок
выходит и бот перестаёт выдавать коды, доступ действительно прекращается.

```
/rent cs2-prime 24       # аренда 24 ч, один арендатор
/rent cs2-prime 24 3     # аренда 24 ч, ТРИ арендатора на аккаунт сразу
/rent cs2-prime 0        # продавать навсегда (аккаунт не вернётся)
/rents                   # кто арендует сейчас, сколько осталось, занятость слотов
/endrent 12              # прекратить аренду досрочно
/free 12                 # вернуть в оборот после проверки
```

Цикл аккаунта: `free → rented → (срок вышел) → free` — и снова. Счётчик
`сдавался раз` виден в уведомлениях.

### Несколько арендаторов на один аккаунт (слоты)

`/rent cs2-prime 24 3` — три человека пользуются одним аккаунтом одновременно.

**Это работает только в офлайн-режиме Steam.** Активная игровая сессия у
аккаунта может быть только одна: если арендаторы останутся онлайн, они будут
выбивать друг друга. Поэтому при `слоты > 1` бот добавляет в сообщение
инструкцию: войти → «Перейти в автономный режим» → играть. Без этого схема
рассыпется, и вы получите возвраты.

Логика распределения:
- слоты занимаются на уже начатом аккаунте, а пустые не трогаются, пока есть
  место — пароль расходится по меньшему числу аккаунтов;
- аккаунт помечается `rented` только когда заняты **все** слоты;
- освободился один слот — аккаунт снова доступен, остальные арендаторы не
  трогаются;
- продажа навсегда всегда занимает аккаунт целиком: `/rent товар 0 3` бот
  отклонит.

**Напишите в описании лота, что аккаунт общий и нужен офлайн-режим.** Это не
формальность: покупатель должен понимать, что берёт. Если лот обещает личный
аккаунт, а слотов три — это уже обман, и возвраты будут заслуженными.

Если включить `RENTAL_MAINTENANCE=true`, аккаунт после аренды уйдёт не в
`free`, а в `maintenance` и будет ждать вашей команды `/free` — время сменить
пароль. Без этого он сдаётся следующему сразу.

### Что честно знать про риски аренды

- **Арендатор может отвязать Steam Guard**, пока у него есть доступ, — и тогда
  аккаунт уйдёт к нему. Код от этого не защитит, это риск модели. Смягчается
  залогом, проверкой рейтинга покупателя и коротким сроком аренды.
- **Пароль виден в чате сделки.** Арендатор может его записать, но без кода
  Guard войти позже не сможет. Если всё же хотите перестраховаться — включите
  `RENTAL_MAINTENANCE` и меняйте пароль между арендами.
- **Steam не любит шаринг аккаунтов** — риск блокировки лежит на вас.

## Запуск

```bash
pip install -r requirements.txt
cp .env.example .env        # заполнить
python bot.py
```

Ключ шифрования настраивать не нужно: при первом запуске бот создаст
`data/.fernet_key` сам.

### .env

```
BOT_TOKEN=123456:AA...        # от @BotFather
ADMIN_IDS=111111111           # ваш Telegram ID (через запятую, если несколько)
BOT_USERNAME=my_shop_bot      # без @, нужен для ссылок вида t.me/bot?start=TOKEN
FERNET_KEY=                   # можно пусто: создастся сам
DB_PATH=data/store.db
PLAYEROK_COOKIES=             # пусто = ручной режим
PLAYEROK_USER_AGENT=
PLAYEROK_PROXY=               # опционально, если IP сервера не проходит
```

## Как пользоваться

**Продавец:** `/add` → прислать maFile → `login:password` → название товара.
`/stock` — остатки. `/link` — связать лот с товаром. `/issue cs2-prime` —
выдать вручную, бот вернёт ссылку для покупателя.

**Покупатель:** переходит по ссылке (или присылает код выдачи текстом) →
получает данные аккаунта, кнопку «новый код Guard» и maFile на скачивание.
`/code` — коды по всем его аккаунтам.

## Что здесь есть и чего нет

| Модуль | Статус |
|---|---|
| `steam_guard.py` | Готово. TOTP-алгоритм Steam, валидация maFile |
| `db.py` | Готово. Атомарный резерв (два заказа не заберут один аккаунт) |
| `crypto.py` | Готово. maFile и пароли шифруются в БД |
| `bot.py` | Готово. Админка, выдача, обновление кодов |
| `playerok.py` | Готово. Слушатель событий в отдельном потоке |

## Подключение к Playerok

Официального API у площадки нет. Используется неофициальная библиотека
[PlayerokAPI](https://github.com/alleexxeeyy/PlayerokAPI) — она работает не
по токену, а **по cookies живой сессии браузера**.

**Логин с сервера автоматизировать нельзя** — Playerok присылает код
подтверждения, его всё равно вводить руками. Поэтому схема такая:
cookies снимаются в браузере на вашем компьютере, а сервер их только
использует и проверяет.

### Шаг 1. Снять cookies (локально, на своём ПК)

```bash
pip install playwright && playwright install chromium
python tools/get_cookies.py
```
Откроется браузер → войдите в аккаунт → Enter в терминале. Скрипт напечатает
готовые `PLAYEROK_COOKIES` и `PLAYEROK_USER_AGENT`.

Вручную то же самое: F12 → Application → Cookies → `https://playerok.com`,
скопировать `token` и `__ddg3` в формате `__ddg3=ЗНАЧЕНИЕ;token=ЗНАЧЕНИЕ`,
а User-Agent взять из консоли (`navigator.userAgent`). Скрипт просто не даёт
ошибиться с User-Agent — он **обязан** совпадать с браузером, где вы вошли.

### Шаг 2. Проверить cookies на сервере

Скопируйте обе строки в `.env` на сервере и запустите **там**:

```bash
docker compose run --rm bot python tools/check_session.py
```

Должно ответить «Сессия рабочая» и показать ваш ник. Это важный шаг: `token` —
обычный JWT и переносится между IP, но защита площадки может не пропустить
IP датацентра, хотя с домашнего всё работало. Если увидите антибот —
пропишите `PLAYEROK_PROXY` в `.env`.

Как это работает: бот слушает событие «предмет оплачен», находит на складе
аккаунт для этого лота и **сразу пишет в чат сделки логин, пароль и код
Steam Guard**. Покупателю никуда переходить не нужно.

Код Guard живёт 30 секунд, поэтому одной отправки мало. Бот слушает чат
сделки: если покупатель напишет «код» (или «code», «guard», «а где код?»),
он тут же пришлёт свежий. Работает сколько угодно раз.

Что важно в этой логике:
- Аккаунт помечается проданным **только после успешной отправки**. Если
  сообщение не ушло — он остаётся зарезервированным, а вам придёт алерт.
- Ответы на «код» ограничены 5 секундами на чат: спам не устроит нам
  лимит запросов к площадке.
- Свои же сообщения игнорируются, иначе бот ответит сам себе и зациклится.
- Длинные сообщения (>60 символов) бот не трогает: это обычно жалоба
  или вопрос, где нужен живой ответ, а не автомат.

Telegram-ссылка (`/issue`) остаётся как запасной путь — например, если
чат сделки недоступен или нужен сам файл maFile.

### Связка лотов со складом

Бот должен понимать, какой лот какому товару соответствует:

```
/link 019abc-def-123 = cs2-prime     # ID лота приходит в уведомлении о продаже
/links                                # посмотреть все связки
```

Если связки нет, бот пробует сопоставить лот по **точному совпадению
названия** с товаром на складе. Работает, но ID надёжнее: название лота
вы можете переименовать в любой момент, и автовыдача молча сломается.

Пока `PLAYEROK_COOKIES` пуст, бот работает в ручном режиме: `/issue` даёт
ссылку, вы кидаете её покупателю сами.

## Безопасность

- maFile = shared_secret + identity_secret. Это ключи от аккаунта: кто держит
  файл, тот генерирует коды входа. Пароли и maFile в базе зашифрованы.
- **Ключ шифрования**: если `FERNET_KEY` не задан, бот создаст `data/.fernet_key`
  при первом запуске. Это удобно, но защищает слабее — ключ лежит рядом с базой,
  и утечка всего каталога `data/` вскроет и то, и другое. Такой режим спасёт от
  случайно опубликованной базы, но не от доступа к серверу. Нужно надёжнее —
  задайте `FERNET_KEY` переменной окружения и храните отдельно.
- **Сделайте копию ключа** (`data/.fernet_key` или значение `FERNET_KEY`).
  Потеряете — склад не расшифровать ничем, включая бэкапы базы.
- Бот удаляет сообщение с паролем из чата админа после добавления.
- Кнопки обновления кода проверяют `buyer_id` — чужой токен не сработает.
- Резерв делается одним `UPDATE ... WHERE status='free' RETURNING` —
  race condition при параллельных заказах исключён.

## Оговорки

- **Библиотека неофициальная.** Playerok может в любой момент поменять API,
  и автовыдача сломается без предупреждения. Ручной режим (`/issue`) —
  ваш запасной вариант, держите его в голове. Cookies тоже протухают:
  бот напишет вам в Telegram, если сессия отвалилась.
- Автоматизация аккаунта может нарушать правила площадки — риск блокировки
  продавца лежит на вас.
- Продажа аккаунтов Steam противоречит Steam Subscriber Agreement: аккаунты
  могут быть заблокированы или отозваны через восстановление. Это риск
  бизнес-модели, а не кода.
